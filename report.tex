\section{Overview}
\begin{itemize}
  \item \textbf{Objective:}
  The goal for this project is to gain a level of understanding of how certain quantization schemes impact a model in a few different ways. (All of these will be done in comparison to the baseline model):
  \begin{itemize}
      \item performance
      \item memory consumption
      \item throughput
  \end{itemize}
  To do so, I'll apply FP8, INT8, INT4, and FP4 quantization schemes to a small decoder-only model and a small MoE model using \texttt{llm-compressor}.
  \item \textbf{Models:}
  The models in question for this weekend+future week's study are:
    \begin{itemize}
      \item \href{https://huggingface.co/meta-llama/Llama-3.2-1B-Instruct}{Meta-Llama-3.2-1B-Instruct} for the decoder-only model

    \end{itemize}

\section{MoE-Specific Observations}

\subsection{Layer Ignore Strategy}

When quantizing MoE models, certain layers must be excluded from quantization to preserve model quality. Two layers were kept at full precision in all Qwen MoE quantization runs:

\begin{itemize}
  \item \textbf{\texttt{lm\_head}} — The final projection layer that maps hidden states to vocabulary logits. Every token prediction flows through this layer, making it disproportionately sensitive to quantization error. Since it represents a negligible fraction of total model parameters, the cost of keeping it in FP16 is effectively zero.

  \item \textbf{\texttt{re:.*mlp.gate\$}} — The MoE router weights. For each token, the router computes scores across all 60 experts and selects the top 4 for activation. These weights are extremely small (a linear projection of $2048 \times 60 \approx 120$K parameters per layer), but quantization errors here can change \emph{which experts are selected} entirely — a token intended for expert 12 might be misrouted to expert 37. This cascading misrouting degrades quality far more than slightly imprecise arithmetic within the correct expert. The compression savings from quantizing these tiny layers would be negligible anyway.
\end{itemize}

Note that Qwen1.5-MoE-A2.7B does not use a \texttt{shared\_expert\_gate} — the shared expert is always active (unconditionally added to every token's computation), so no additional gate ignore is needed.

\subsubsection{Why Not Ignore Entire Early Layers?}

An alternative strategy would be to skip quantization of early transformer layers entirely, based on the intuition that early layers learn more foundational representations. However, GPTQ's calibration-based approach already compensates for per-layer sensitivity by optimizing quantization parameters using the Hessian of each layer's activations. Ignoring entire layers forfeits the compression opportunity without meaningful quality benefit. The gate-only strategy is more surgical: protect the small, fragile routing decisions and quantize everything else.
